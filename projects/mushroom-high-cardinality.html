Handling High Cardinality and False Negatives in Mushroom Classification

GitHub Repo: https://github.com/sam-tritto/mushrooms-high-cardinality

Introduction: The Cost of a False Negative

In machine learning, we often evaluate models using clean metrics like accuracy or F1 score. We celebrate a 99% accuracy rate, assuming the model is safe for production. However, in high-stakes domains, all errors are not created equal. If I am building a model to classify mushrooms as edible or poisonous, a false positive (calling an edible mushroom poisonous) results in a wasted meal. A false negative (calling a poisonous mushroom edible) can result in organ failure or death. It is the ultimate high-stakes classification problem.

To make matters worse, real-world data is rarely as simple as the toy datasets we study. When we move from the clean, classic datasets of the 1980s to modern, noisy datasets, we encounter high-cardinality categorical features - features with dozens of unique categories - alongside massive gaps of missing data. If we rely on naive encoding methods like One-Hot Encoding, we explode our feature space, create sparse matrices, and drown our models in noise. If we are not careful, these representations introduce blind spots that lead directly to fatal false negatives.

In this tutorial, I will walk through a full comparative analysis of the classic UCI Mushroom dataset (1987) and the modern UCI Secondary Mushroom dataset (2023). I will explore why naive encodings fail, how gradient boosting libraries like LightGBM and CatBoost handle high cardinality natively, and I will dissect the exact rows where these state-of-the-art models failed. Finally, I will demonstrate three production-grade mitigations to eliminate false negatives entirely.

1. The Two Datasets

To understand the challenge of high cardinality, I compared two distinct eras of mushroom classification data:

Metric Classic Mushroom Dataset (1987) Secondary Mushroom Dataset (2023)
UCI ID 73 848
Row Count 8,124 61,069
Feature Count 22 (all nominal) 20 (nominal + metrical)
Max Cardinality 12 (gill-color) 36 (cap-color, habitat, etc.)

The secondary dataset is much more challenging. It was synthesized from 173 real species to mimic real-world complexity. It mixes metrical features (such as cap-diameter, stem-height, and stem-width) with high-cardinality categorical features (such as habitat, season, and cap-color, which has 36 unique values). It also contains vast swathes of missing data, forcing us to handle missingness as a first-class citizen rather than dropping rows.

2. The Encoding Spectrum

Before training models, I had to decide how to encode these categorical variables. Traditional linear models cannot digest raw strings, and gradient boosters require numeric representation. The choice of encoding represents a trade-off between dimensionality, information retention, and target leakage:

One-Hot Encoding (OHE) OHE creates a binary column for every unique category. For low-cardinality features, this is safe and effective. But for high-cardinality features like cap-color, it blows up the feature space. In our secondary dataset, OHE expanded our 20 input features into 128 columns. This introduces massive sparsity and memory bloat, which degrades the performance of linear models.
Target Encoding (TE) Target Encoding replaces each category value with the mean of the target variable for that category. It keeps the feature space compact (1 column per categorical feature) but introduces a major risk of target leakage. If the encoder peeks at the target value of the row it is trying to encode, the model will overfit during training and fail on test data. Scikit-learn's target encoder uses cross-fitting to reduce this leakage, but it remains a soft defense. 
# Naive Target Encoding Pipeline with scikit-learn
preprocessor_te = ColumnTransformer([
    ('num', StandardScaler(), num_cols_s),
    ('cat', TargetEncoder(random_state=RANDOM_STATE), cat_cols_s),
])

logreg_te_pipe = Pipeline([
    ('pre', preprocessor_te),
    ('clf', LogisticRegression(max_iter=500, C=1.0, random_state=RANDOM_STATE))
])
logreg_te_pipe.fit(X_s_train, y_s_train)
CatBoost's Secret Weapon: Ordered Target Encoding CatBoost solves the leakage problem by design. It shuffles the training dataset and, for each row, computes the target statistics using only the rows that precede it in the shuffle. This mimicking of a time-series or online learning flow prevents the model from peeking into the future, eliminating target leakage while preserving all the predictive power of high-cardinality features. 
# CatBoost data preparation and training
# Note: CatBoost requires categorical NaNs to be filled with strings
def prepare_catboost(X, cat_cols):
    df = X.copy()
    for col in cat_cols:
        df[col] = df[col].fillna('NA').astype(str)
    return df

X_s_train_cb = prepare_catboost(X_s_train, cat_cols_s)
X_s_test_cb  = prepare_catboost(X_s_test,  cat_cols_s)
cat_idx_s = [X_s_train_cb.columns.get_loc(c) for c in cat_cols_s]

cb_s = CatBoostClassifier(
    iterations=500,
    learning_rate=0.05,
    depth=6,
    cat_features=cat_idx_s,
    random_state=RANDOM_STATE,
    verbose=0
)
cb_s.fit(X_s_train_cb.values, y_s_train.values)

4. Anatomy of the Failures: False Negative Analysis

Despite accuracy rates exceeding 99.9%, a single mistake can be fatal. I isolated the exact rows where the models failed. The findings are a cautionary tale of over-reliance on single features and missing data blind spots.

The Odor Trap in the Classic Dataset (Row 5237)

Both Logistic Regression and CatBoost misclassified Row 5237 as edible (a false negative), while LightGBM correctly identified it as poisonous.

The culprit was the `odor` feature. In the classic dataset, the rule odor = none (n) implies edible is true for approximately 99% of samples. As a result, the models placed almost all their predictive weight on this single feature. Row 5237 represents a poisonous Amanita species that has no odor. Because it lacked odor, bruising, and had white gills and a white spore print, the model fell into the trap:

Model Predicted Probability P(poisonous) for Row 5237 Outcome
Logistic Regression + OHE 0.4533 False Negative (called Edible) ⚠️
CatBoost (ordered TE) 0.4573 False Negative (called Edible) ⚠️
LightGBM (native cats) 0.9999 True Positive (called Poisonous) ✅

The Lesson: A single, near-deterministic feature creates a fragile rule. If a poisonous mushroom mimics a safe odor, the model will confidently kill the user. LightGBM survived because its tree splits identified a complex interaction (bulbous stalk-root + clustered population + leaf habitat) that overrode the odor prior. CatBoost's ordered target encoder encoded `odor = n` with such a high edible score that the rest of the tree could not override it.

The Missing Data Blind Spot in the Secondary Dataset

In the secondary dataset, LightGBM and CatBoost each missed exactly one row - but they missed different rows (LightGBM missed Row 7154; CatBoost missed Row 58808).

When I inspected these rows, I found they were among the most data-sparse examples in the entire test set. They had missing values (NaN) for 5 to 6 of the most informative features:

  • `stem-root` = NaN
  • `stem-surface` = NaN
  • `veil-type` = NaN
  • `veil-color` = NaN
  • `spore-print-color` = NaN
  • `gill-spacing` = NaN

When almost all key features are missing, the models are forced to classify based on weak residual signals (like cap-diameter and season) and ultimately guessed wrong.

5. Production-Grade Mitigations

We cannot deploy these models to production as-is. We need safeguards to catch the rare but dangerous false negatives. I implemented and verified three strategies to address these failure modes.

Strategy A: Conservative OR-Voting Ensemble

Since different models have different blind spots (e.g., LightGBM caught the classic odor trap while CatBoost missed it), we can build an ensemble. Instead of using majority voting, we use a conservative OR-vote: if any individual model predicts poisonous, the ensemble predicts poisonous.

When I applied this to the classic test set, the OR-vote eliminated the false negative on Row 5237 while maintaining 100% overall accuracy:

# Conservative OR-vote ensemble logic
preds_logreg = ohe_pipe.predict(X_c_test)
preds_lgb    = lgb_c.predict(X_c_test_lgb)
preds_cb     = cb_c.predict(X_c_test_cb.values)

# If any model predicts 1 (poisonous), output 1
preds_or = (np.stack([preds_logreg, preds_lgb, preds_cb], axis=1).sum(axis=1) >= 1).astype(int)

Strategy B: Missingness-Count Feature

To address the sparse row failures on the secondary dataset, I engineered a new feature, `n_missing_cats`, which counts the number of missing categorical values in a row. The absence of details (such as veil color or stem surface) is itself a strong biological signal: rare or hard-to-describe species are more likely to have missing features and are often poisonous.

While adding this feature did not fully eliminate the single false negative (it went from 1 to 1 for LightGBM, and 1 to 2 for CatBoost due to shifting splits), it successfully reduced overall false positives and allowed the trees to establish explicit splits based on data sparsity.

# Feature engineering missingness count
def add_missingness_feature(X, cat_cols):
    df = X.copy()
    df['n_missing_cats'] = df[cat_cols].isnull().sum(axis=1)
    return df

X_s_train_m  = add_missingness_feature(X_s_train, cat_cols_s)
X_s_test_m   = add_missingness_feature(X_s_test,  cat_cols_s)

Strategy C: Predict-With-Abstain (The Reject Option)

The most robust defense in a safety-critical application is the **Reject Option**. If the model's confidence is low, it should refuse to make a prediction and instead flag the sample for expert mycologist review. I implemented a wrapper function that abstains if the maximum predicted probability falls below a set threshold (e.g., 80%):

def predict_with_abstain(model, X, threshold=0.80):
    probs = model.predict_proba(X)
    max_probs = np.max(probs, axis=1)
    preds = np.argmax(probs, axis=1)
    
    # -1 represents "Abstain / Refuse to predict"
    final_preds = np.where(max_probs >= threshold, preds, -1)
    return final_preds

When evaluated at an 80% confidence threshold, the results on our test sets were clear:

  • CatBoost (Classic): Abstains on 2 rows (0.1% of test set). On the remaining rows: 0 False Negatives, 0 False Positives.
  • LightGBM (Secondary): Abstains on 3 rows (0.02% of test set). On the remaining rows: 0 False Negatives, 0 False Positives.
  • CatBoost (Secondary): Abstains on 4 rows (0.03% of test set). On the remaining rows: 0 False Negatives, 0 False Positives.

By allowing the model to say "I don't know" on just a handful of highly uncertain, data-sparse rows, we completely wiped out the dangerous false negatives without impacting performance on the remaining 99.9% of the dataset.

Key Takeaways

  1. OHE is a poor choice for high cardinality. Naive OHE degrades the performance of linear models and bloats memory. Use native categorical splitting (LightGBM) or Ordered Target Encoding (CatBoost) instead.
  2. Accuracy hides critical failures. A model with 99.9% accuracy can still have a blind spot. Always inspect your false negatives, especially in safety-critical domains.
  3. Be wary of "magic bullet" features. Features like odor that are 99% predictive can act as traps, causing models to ignore other critical features.
  4. Sparsity is a signal. Gaps of missing values represent a lack of information that can lead to incorrect guesses. Tracking missingness explicitly using a count feature can provide valuable signal.
  5. Abstention is a required fallback. In high-stakes environments, always implement a reject option. Refusing to predict on low-confidence rows saves lives.
