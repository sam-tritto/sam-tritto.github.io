Introduction: A New Tabular Paradigm

In tabular machine learning, XGBoost, LightGBM, and CatBoost have long ruled the roost. If you wanted to classify rows of data, you collected training samples, engineered features, and ran gradient boosting backpropagation to optimize decision trees. However, a new paradigm is shifting this workflow: Tabular Foundation Models. Google's TabFM (Tabular Foundation Model) is a transformer-based architecture (detailed in their Introducing TabFM research blog and hosted on the google-research/tabfm GitHub repository) capable of performing in-context learning (ICL) directly on tabular datasets.

To put this technology to the test, I set up a high-stakes prediction task: classifying case fatalities (Fatal (Y/N)) from real historical shark encounter reports along the coastlines of Hawai'i. This project runs a head-to-head battle between a zero-shot, pre-trained TabFM model and a supervised CatBoost Classifier.

MacBook Footprint Warning While TabFM is simple to load and install natively through HuggingFace (using the weights from google/tabfm-1.0.0-pytorch), the pre-trained weights require a hefty 6.5GB. Fitting this footprint onto my MacBook's unified memory took a noticeable toll on local resources, making it a heavy baseline for quick local workflows, even though the configuration requires zero custom tokenizers.

1. The Raw Data: Messy Narratives from Hawai'i

The data comes from the official Global Shark Attack File (GSAF), specifically utilizing the primary GSAF historical log spreadsheet. Instead of clean, standardized categories, the GSAF is notorious for unstructured, high-cardinality text narratives. For example, rather than a simple label like "Surfing", the Activity column contains verbose descriptions of the incident circumstances:

  • "Surfing, fell off board, bit on leg"
  • "Wading / standing in 3 ft of water"
  • "Spearfishing / free diving with catch"

I cleaned and prepared a localized subset consisting of incidents in Hawai'i, predicting whether the encounter was fatal (1) or survived (0) based on four features: Activity, Time of Day, Month, and Cleaned Species (derived from raw text descriptions).

2. Why These Two Libraries?

This messy, textual dataset provides an ideal sandbox for contrasting the two paradigms:

  • CatBoost: Standard gradient boosting algorithms struggle with raw text unless you pre-process it into token vectors. CatBoost, however, includes a native text processing engine. It builds dictionaries and tokenizers on the fly, directly digesting the unstructured Activity strings alongside categorical predictors without manual feature engineering pipelines.
  • TabFM: As a foundation model, TabFM utilizes an alternating row/column attention mechanism. It performs In-Context Learning by reading the training records as a "prompt" inside its transformer attention window. In a single forward pass, it makes zero-shot predictions on the evaluation set without updating a single weight or running backpropagation.

The Power of Semantic Pre-Training

Why is TabFM uniquely suited for this messy text? Unlike standard tree classifiers that treat category tokens as arbitrary IDs, TabFM is built on top of a pre-trained language transformer. Because it has been pre-trained on tabular data containing rich text descriptions, the model inherently understands the semantic meaning of terms like "surfing", "wading", and "spearfishing", as well as the anatomical contexts of injuries. It leverages these semantic text embeddings and language priors directly, which explains why it performs well out of the box on unstructured categorical narratives without needing custom feature tokenization or vectorizers.

3. The In-Context Scaling Challenge

For experienced researchers, the primary architectural challenge with TabFM is context window scaling. Because TabFM processes the tabular matrix as a prompt, memory and compute scale dramatically with the size of the training pool. To operate within its attention limits, we cannot feed a massive dataset to the model. I implemented a stratified subsampling step to select a representative context dataset of up to 500 rows, preserving the target's class distribution:

# Stratified context preparation for TabFM
from sklearn.model_selection import train_test_split

def prepare_tabfm_context(X_train, y_train, max_rows=500, random_state=42):
    if len(X_train) <= max_rows:
        return X_train, y_train
    
    test_size = max_rows / len(X_train)
    _, X_context, _, y_context = train_test_split(
        X_train, y_train, 
        test_size=test_size, 
        stratify=y_train, 
        random_state=random_state
    )
    return X_context, y_context

4. Modeling Ensembles & Codes

Google TabFM Inference

Loading the weights and performing predictions is remarkably straightforward using the PyTorch-based HuggingFace wrapper:

from tabfm import TabFMClassifier
from tabfm import tabfm_v1_0_0_pytorch as tabfm_v1_0_0

# Load weights natively from HuggingFace
tabfm_model = tabfm_v1_0_0.load()
tabfm_clf = TabFMClassifier(model=tabfm_model)

# Fit behaves as an in-context setup step
tabfm_clf.fit(X_context, y_context)

# Generate zero-shot probabilities
tabfm_probs = tabfm_clf.predict_proba(X_test)[:, 1]

Supervised CatBoost Classifier & Early Stopping

To train the supervised CatBoost Classifier without overfitting on this small, localized dataset, I implemented an early stopping strategy. First, I performed an 80/20 train-validation split on the training set to monitor the model's generalization performance and find the optimal iteration count without data leakage. Once the validation loss (measured in AUC) ceased to improve for 100 consecutive rounds, the training stopped. Finally, I re-trained the model on the full training set (utilizing 100% of the available training data) up to that optimal iteration count.

Here is the implementation of the validation split and early stopping configuration, declaring the Activity feature as raw text:

from catboost import CatBoostClassifier, Pool
from sklearn.model_selection import train_test_split

# Declare raw text columns and categoricals
text_cols = ['Activity']
cat_cols = ['Time of Day', 'Month', 'Cleaned Species']

# Perform an 80/20 train-validation split on the training set to prevent leakage
X_train_sub, X_val, y_train_sub, y_val = train_test_split(
    X_train, y_train, test_size=0.2, random_state=42, stratify=y_train
)

train_sub_pool = Pool(X_train_sub, y_train_sub, cat_features=cat_cols, text_features=text_cols)
val_pool = Pool(X_val, y_val, cat_features=cat_cols, text_features=text_cols)
test_pool = Pool(X_test, cat_features=cat_cols, text_features=text_cols)

# Train with early stopping (100 rounds) to identify the optimal iteration
cat_model_es = CatBoostClassifier(
    iterations=1000,
    learning_rate=0.05,
    eval_metric='AUC',
    random_seed=42,
    early_stopping_rounds=100,
    verbose=100
)

# Fit on sub-train and evaluate on the validation pool
cat_model_es.fit(train_sub_pool, eval_set=[train_sub_pool, val_pool])

best_iteration = cat_model_es.get_best_iteration()
best_val_score = cat_model_es.get_best_score()['validation_1']['AUC']

With the optimal iteration count (111) established, I re-trained the final model on the complete training set (combining both the sub-train and validation sets) to maximize the training data footprint before evaluating on the held-out test set:

# Re-train the final model on the full training set for 111 iterations
train_pool = Pool(X_train, y_train, cat_features=cat_cols, text_features=text_cols)

cat_model = CatBoostClassifier(
    iterations=best_iteration + 1,  # 111 iterations
    learning_rate=0.05,
    eval_metric='AUC',
    random_seed=42,
    verbose=False
)
cat_model.fit(train_pool)

# Predict probabilities on the test set
cat_probs = cat_model.predict_proba(test_pool)[:, 1]

The Imbalanced Class Collapse

The test dataset consists of 63 encounters: 53 survived and 10 fatal (a 16% fatality rate). Using the standard decision threshold of 0.5, CatBoost predicts "Survived" for every single row. Because the local dataset is small and highly imbalanced, the supervised optimizer collapsed to the majority class to maximize overall accuracy (84%).

In contrast, TabFM's zero-shot pre-trained priors prevented it from collapsing. Even without class weight adjustment, it correctly isolated 2 of the 10 fatal attacks (achieving 20% recall and 40% precision), demonstrating how foundation models can be more robust to class imbalance out of the box.

6. The Calibration Trade-off

While ROC AUC indicates ranking strength (where CatBoost led with 0.7321 over TabFM's 0.6755), probability calibration is critical for real-world risk management. A model is calibrated if a predicted probability of 20% maps to an actual event rate of 20% in the real world.

In-context foundation models are notoriously poorly calibrated. Because they make predictions in a single forward pass over synthetic training patterns, their raw outputs do not scale to match the baseline distribution of the localized test environment. CatBoost, by contrast, is directly optimized to match the target statistics of the training pool, aligning its probabilities closer to the true physical rate. For researchers using zero-shot models in critical decision systems, post-processing calibration (such as Platt scaling or isotonic regression) is a non-negotiable step.

Conclusion: When to Choose What?

For quick deployments on datasets with heavy, messy categorical text narratives, Google's TabFM represents an exciting paradigm shift, saving days of feature engineering and training pipeline design. However, if resource footprints on edge devices are a constraint, or if probability calibration is vital for safety-critical thresholds, the native text-processing engines of supervised gradient boosters like CatBoost remain the gold standard.
