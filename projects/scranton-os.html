Orchestrating Cloud Ops with a Multi-Agent Team modeled after The Office

What happens when you combine the chaotic energy of Dunder Mifflin with the Google Antigravity SDK and FastAPI? You get ScrantonOS—a persistent, multi-agent orchestration engine and web interface built to handle corporate cloud operations, system diagnostics, and compliance workflows.

In a typical engineering environment, routing cloud requests, reviewing IAM permissions, analyzing budgets, and managing data deletion is handled by disjointed CLI scripts or tickets. ScrantonOS replaces this with a cohesive, Natural Language processing routing loop that delegates technical tasks to a team of agents, each possessing the distinct personality (and functional competence) of characters from The Office.

Architecture Overview

ScrantonOS operates as a hub-and-spoke system. Natural language requests are received by the orchestrator agent and then routed dynamically to subject matter experts based on intent classification. If a task requires multiple steps, the orchestrator can chain tasks together, creating an automated pipeline.

Multi-Agent Routing & The Orchestration Loop

When a request is submitted, Michael Scott classifies it into one of over 12 operational categories (such as SRE, FinOps, IAM, and Data Purge). Below is a snippet of Michael's intent classification instructions: 

# Intent Classification Categories
- **sre**: Cloud logs, crashes, errors, system failures, uptime
- **finops**: Billing, costs, spending, budget, GPU usage
- **iam**: Access requests, permissions, roles, security clearance
- **purge**: Data deletion, GDPR compliance, PII removal
- **report**: Generate a summary, artifact, formatted output

Once classified, the session payload is forwarded to the corresponding agent. The workflow supports multi-turn conversations, caching state dynamically.

Human-in-the-Loop (HITL) Gatekeeping

Not all operations should be automated. Destructive tasks (like DELETE operations) or security elevations (like IAM role changes) require human oversight. ScrantonOS enforces this via a compliance mechanism managed by Toby Flenderson:

  1. The agent identifies a high-risk operation and generates a pending workflow token.
  2. The orchestration engine suspends execution and routes the token to Toby.
  3. Toby blocks the workflow and surfaces an approval request on the Web Interface (or returns a pending token via the REST API).
  4. The user must review, add a reason, and explicitly click "Approve" or "Reject" to resume or abort the operation.

This ensures compliance is maintained without halting the chat interface for other non-destructive queries.

Cloud Scaling and Persistence Modes

To support local development and production deployments, ScrantonOS implements a clean database abstraction:

  • DEV Mode: Relies on local CSV files for conversation logging, audit logs, and session management. This enables fast offline iteration without needing cloud database connections.
  • PROD Mode: Auto-switches to Google Cloud Firestore. In production, session documents, thread states, and compliance logs are written to Firestore collections. Secrets are pulled securely from GCP Secret Manager at runtime rather than being hardcoded.

Local Setup and Deployment

Testing ScrantonOS locally is straightforward using Docker: 

# Build and run the multi-agent container
docker-compose up --build

# Generate synthetic billing, logs, and ticket data for testing
python scripts/generate_synthetics.py

For cloud environments, the server is fully containerized and compatible with serverless engines like Google Cloud Run. WebSocket support ensures real-time streaming of agent responses back to the browser console.
