Building and Querying a Climate-Tourism Knowledge Graph in BigQuery

GitHub Repo: https://github.com/sam-tritto/climate-tourism-knowledge-graph

Introduction: Reconciling Sensor and Topological Data

Climate change is restructuring global economic activity. Among the most direct impacts is the shifting geography of summer tourism. As Southern European destinations (like Spain and Italy) experience increasingly regular summer heatwaves exceeding comfortable ranges, consumer travel searches and flights show signs of redirecting toward cooler high-latitude regions (like Norway and Sweden).

To analyze this phenomenon at scale, I built a Climate-Tourism Knowledge Graph. Instead of extracting spatial and temporal tables to a specialized graph database (which introduces data duplication and maintenance overhead), I compiled and queried the graph directly inside Google BigQuery. BigQuery now natively supports the SQL/PGQ standard (Property Graph Queries), allowing data teams to perform complex multi-hop path traversals using standard SQL.

This tutorial details how to prepare the node and edge datasets, build the property graph schema using DDL, run GQL (Graph Query Language) pattern matches, fit comfort tipping points, and analyze Heathrow flight routing patterns during peak summer anomalies.

Target Audience & Key Concepts
  • For Beginners: You will see how graph modeling simplifies relationships that would otherwise require writing nested, multi-table SQL joins.
  • For Experienced Researchers: We walk through a piecewise linear regression (hinge model) used to calculate comfort tipping point thresholds, and examine the technical performance bottlenecks of running graph engines inside a columnar data warehouse.

1. The Raw Data and Where It Comes From

I aggregated and synchronized three distinct datasets to capture meteorological observations, consumer digital intent, and the physical transport network:

A. NOAA Global Summary of the Day (GSOD)

Meteorological records are queried directly from the BigQuery public dataset bigquery-public-data.noaa_gsod.gsod2025. This dataset hosts daily sensor observations from weather stations globally. I extracted the summer period (June to August 2025) and mapped station indices to country codes. I converted temperature readings from Fahrenheit to Celsius and resolved missing codes. To save query scanning costs, I strictly filtered the table by year using _TABLE_SUFFIX = '2025'.

B. Google Trends

To measure consumer booking intent, I used the pytrends library to pull historical weekly search interest volumes. I extracted search queries from origin markets (Germany and the United Kingdom) targeting Southern destinations (Spain, Italy) and Northern alternatives (Norway, Sweden). Search volumes are scaled from 0 to 100 based on relative local popularity.

C. OpenFlights Topography

The transport layer represents physical routes. I downloaded OpenFlights airports and route files, filtering for flights operating within the European network. I aggregated route tables by source-destination airport pairs to construct connection frequencies (labeled airline_count), representing route capacity.

2. Designing the Graph Schema in BigQuery

A property graph maps entities as nodes and relationships as edges. The schema uses three node tables and four edge tables:

  • Node Tables:
    • countries: Keyed by country_code.
    • airports: Keyed by IATA airport code.
    • weather_stations: Keyed by NOAA station ID.
  • Edge Tables:
    • LocatedIn (Airport → Country): Connects physical airports to their countries.
    • LocatedIn (WeatherStation → Country): Links sensor stations to countries. Note that in GQL we declare the label as StationLocatedIn to distinguish it from airport locations.
    • FliesTo (Airport → Airport): Represents directional flight capacity.
    • SearchesFor (Country → Country): Links origin search markets to destination countries. This edge carries weekly properties: search_volume, dest_max_temp_c, and dest_temp_anomaly.

I executed the following Data Definition Language (DDL) command to compile these elements into a unified property graph structure:

CREATE OR REPLACE PROPERTY GRAPH climate_tourism.tourism_graph
NODE TABLES (
  climate_tourism.countries KEY (country_code) LABEL Country,
  climate_tourism.airports KEY (iata) LABEL Airport,
  climate_tourism.weather_stations KEY (station_id) LABEL WeatherStation
)
EDGE TABLES (
  climate_tourism.located_in_airport KEY (iata)
    SOURCE KEY (iata) REFERENCES climate_tourism.airports (iata)
    DESTINATION KEY (country_code) REFERENCES climate_tourism.countries (country_code)
    LABEL LocatedIn,
  climate_tourism.located_in_station KEY (station_id)
    SOURCE KEY (station_id) REFERENCES climate_tourism.weather_stations (station_id)
    DESTINATION KEY (country_code) REFERENCES climate_tourism.countries (country_code)
    LABEL StationLocatedIn,
  climate_tourism.flight_routes KEY (source_airport, destination_airport)
    SOURCE KEY (source_airport) REFERENCES climate_tourism.airports (iata)
    DESTINATION KEY (destination_airport) REFERENCES climate_tourism.airports (iata)
    LABEL FliesTo,
  climate_tourism.search_correlations KEY (origin_country, destination_country, week)
    SOURCE KEY (origin_country) REFERENCES climate_tourism.countries (country_code)
    DESTINATION KEY (destination_country) REFERENCES climate_tourism.countries (country_code)
    LABEL SearchesFor
);

3. Querying with GQL: Visual Matches and Traversal Patterns

Once defined, we query the graph using the standard GRAPH ... MATCH pattern matching syntax. This section highlights five GQL queries designed to trace climate risks across the network. If implemented in standard relational SQL, these queries would require writing numerous complex, nested joins and recursive Common Table Expressions (CTEs).

Query 1: Exposed Flight Corridors

This query identifies flight corridors connecting origin markets directly to destination airports inside countries experiencing extreme summer heatwaves (weekly maximum temperatures exceeding 38°C).

GRAPH climate_tourism.tourism_graph
MATCH (orig:Country)-[s:SearchesFor]->(dest:Country)<-[:LocatedIn]-(ap_dest:Airport)
MATCH (ap_orig:Airport)-[r:FliesTo]->(ap_dest)
MATCH (ap_orig)-[:LocatedIn]->(orig)
WHERE s.dest_max_temp_c > 38.0
RETURN 
  orig.name as origin, 
  ap_orig.iata as origin_airport, 
  dest.name as destination, 
  ap_dest.iata as destination_airport, 
  r.airline_count as route_density, 
  s.dest_max_temp_c as max_temp,
  s.week as week
ORDER BY r.airline_count DESC
LIMIT 5;

Query 2: Multi-Hop Connection Hubs

This query searches for two-leg flight paths connecting an origin country to a heat-stressed destination country via a transit airport located in a third, separate country. It helps researchers evaluate routing hubs exposed to regional disruptions.

GRAPH climate_tourism.tourism_graph
MATCH (orig:Country)-[s:SearchesFor]->(dest:Country)
MATCH (ap_orig:Airport)-[:LocatedIn]->(orig)
MATCH (ap_orig)-[f1:FliesTo]->(ap_hub:Airport)-[:LocatedIn]->(hub:Country)
MATCH (ap_hub)-[f2:FliesTo]->(ap_dest:Airport)-[:LocatedIn]->(dest)
WHERE orig.country_code != hub.country_code 
  AND dest.country_code != hub.country_code
  AND s.dest_max_temp_c > 38.0
RETURN 
  orig.name as origin_country, 
  ap_orig.iata as origin_airport, 
  hub.name as hub_country,
  ap_hub.iata as hub_airport, 
  dest.name as destination_country, 
  ap_dest.iata as destination_airport,
  f1.airline_count as leg1_density,
  f2.airline_count as leg2_density
LIMIT 5;

Query 3: Reciprocal Flight Loops (Cycle Detection)

This query isolates pairs of airports that operate mutual, two-way flight connections within markets actively experiencing temperature anomalies. Detecting cycles is a common graph challenge that is simplified here with bidirectional matches.

GRAPH climate_tourism.tourism_graph
MATCH (a1:Airport)-[f1:FliesTo]->(a2:Airport)-[f2:FliesTo]->(a1)
MATCH (a1)-[:LocatedIn]->(c1:Country)
MATCH (a2)-[:LocatedIn]->(c2:Country)
MATCH (c1)-[s:SearchesFor]->(c2)
WHERE s.dest_max_temp_c > 38.0
RETURN 
  c1.name as country_a, a1.iata as airport_a, 
  c2.name as country_b, a2.iata as airport_b, 
  f1.airline_count as density_a_to_b, 
  f2.airline_count as density_b_to_a;

Query 4: Localized Station-to-Intent Traversal

This traversal links physical ground sensors directly to digital search logs. It traces weather stations reporting local maximum temperatures above 40°C back to the search volume volumes recorded in European origin markets.

GRAPH climate_tourism.tourism_graph
MATCH (orig:Country)-[s:SearchesFor]->(dest:Country)<-[:StationLocatedIn]-(ws:WeatherStation)
WHERE s.dest_max_temp_c > 40.0
RETURN 
  ws.name as weather_station, 
  dest.name as destination, 
  orig.name as origin, 
  s.search_volume as search_volume,
  s.dest_max_temp_c as recorded_temp
ORDER BY s.search_volume DESC
LIMIT 5;

Query 5: Variable-Length Connection Finder

This query runs a transitive path match to retrieve all flight routing paths (ranging from 1 to 3 layovers) from Munich (MUC) to Tenerife (TFS) during a summer heatwave week.

GRAPH climate_tourism.tourism_graph
MATCH (ap_orig:Airport {iata: 'MUC'})-[:FliesTo]->{1,3} (ap_dest:Airport {iata: 'TFS'})
MATCH (ap_orig)-[:LocatedIn]->(c_orig:Country)-[s:SearchesFor]->(c_dest:Country)
MATCH (ap_dest)-[:LocatedIn]->(c_dest)
WHERE s.dest_max_temp_c > 38.0
RETURN ap_orig.iata as origin, ap_dest.iata as destination, s.week as week;

4. Isolating the Climate Tipping Points

Is there a structural comfort threshold where rising summer heat stops attracting tourists and starts driving them away? To isolate this tipping point, I fitted a piecewise linear regression (hinge model) to weekly search volume versus daily maximum temperature. The mathematical model is expressed as:

$$y = \begin{cases} y_0 + k_1(x - x_0) & \text{if } x < x_0 \\ y_0 + k_2(x - x_0) & \text{if } x \ge x_0 \end{cases}$$

Where:

  • \(x\) is the destination weekly maximum temperature (°C).
  • \(y\) is the weekly search interest volume.
  • \(x_0\) is the comfort tipping point temperature threshold.
  • \(y_0\) is the estimated search volume at that tipping point.
  • \(k_1\) is the search elasticity (slope) below the tipping point.
  • \(k_2\) is the search elasticity (slope) above the tipping point.

By fitting this model via non-linear least squares, I isolated two contrasting behaviors:

Destination Tipping Point (\(x_0\)) Elasticity Before (\(k_1\)) Elasticity After (\(k_2\)) Behavioral Interpretation
Spain (Mediterranean) 40.01°C +2.69 -1.45 Comfort limit reached; interest declines rapidly above 40°C.
Norway (Nordic) 33.65°C +0.02 +0.85 Interest is flat in cold zones; interest accelerates as it warms.

Measuring the Redirection Effect

Is this redirection correlated? I ran a temporal correlation analysis comparing weekly Mediterranean temperature anomalies with Nordic search volumes. The correlation coefficient is high (\(r = 0.74\)), confirming that when Southern Europe experiences extreme heat anomalies, tourist interest shifts to cooler high-latitude alternatives.

5. Heathrow Network Case Study

To inspect physical route adjustments, I extracted outbound flights and search intensity originating from London Heathrow (LHR) during a severe heatwave week (July 7, 2025). The analysis confirms that flight capacities to Spain remain dense, but search volumes show accelerated growth toward high-latitude airport nodes as Southern temperatures enter the warning zone.

6. Architectural Trade-offs and the BigQuery Slot Trap

While building a knowledge graph inside BigQuery is highly expressive, researchers must understand the underlying physical limits before selecting it for production architectures.

The Latency and Slot Challenge

BigQuery is a columnar, analytical data warehouse built for high-throughput scans, not transactional pointer-chasing. When a GQL query is executed, BigQuery translates the GQL syntax into a series of hash joins, filter steps, and occasionally recursive loops. This mapping introduces specific bottlenecks:

  • Cold Startup Overhead: Every BigQuery query is submitted as a job, passing through query planning, optimization, and worker scheduling. This introduces a minimum cold startup latency of 1 to 3 seconds, regardless of the dataset size.
  • Slot Consumption: GQL traversals (such as multi-hop routes or cycles) require matching multiple tables. When executing variable-length paths (e.g., [:FliesTo]->{1,3}), the engine evaluates nested self-joins. On large tables, this consumes a massive number of virtual CPU cores (slots), which can cause query starvation or high billing spikes under concurrent workloads.
  • Batch-Only Writes: BigQuery does not support low-latency transactional updates. If you want to modify a single node (like updating a weather station's daily max temperature) or insert a single edge, you must run a DML `INSERT` or `UPDATE` on the underlying tables and rebuild or refresh the property graph schema.
Why BigQuery is NOT a Good Candidate for Agentic/Chatbot Loops Because of the 1 to 3 second job overhead and high slot consumption under concurrent query patterns, BigQuery Graphs should not be used as the direct backend for real-time conversational agents or chatbots. A conversational agent needs sub-second response times (often 50 to 100 milliseconds) to query the graph in multi-step planning loops. Running an agentic loop over BigQuery Graph would introduce lagging, expensive cycles that exhaust database slots quickly. For transactional, low-latency, real-time agent loops, specialized graph databases (like Neo4j or Memgraph) remain the standard.

Why You Should Still Use BigQuery Property Graphs

Despite these transactional limitations, BigQuery Graph is an exceptional tool for analytical workflows:

  • No ETL Pipelines: If your enterprise data already resides in a BigQuery warehouse (e.g., historical flight bookings, CRM details, sensor logs), you do not need to build complex ETL pipelines to sync that data to an external graph database. You build the graph schema on top of your existing tables instantly.
  • Infinite Analytical Scale: While Neo4j can struggle with multi-terabyte aggregations due to memory constraints, BigQuery leverages its massive parallel processing engine to scan and join billions of rows of nodes and edges in seconds.
  • Unified SQL Interface: Data analysts do not need to learn a proprietary database language or maintain separate database instances. GQL is integrated right into standard SQL scripts, making it easy to incorporate graph traversals into existing DBT models and dashboard queries.

Conclusion

Building a Climate-Tourism Knowledge Graph directly in BigQuery provides a unified, powerful way to analyze geographical relationships and environmental sensor data at scale. By combining NOAA weather alerts, Google Trends, and flight topologies, I isolated a comfort tipping point around 40°C for Mediterranean destinations and traced the resulting redirection patterns. While it is not designed for low-latency agentic loops, BigQuery's property graph engine is a game-changer for large-scale, warehouse-native relational analytics.
